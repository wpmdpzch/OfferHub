## 二、需要 PM 决策的遗留问题

### 🔴 P0 - 影响 MVP 开发启动

**问题 1：举报功能的处理流程未定义**

设计文档已加入举报接口和 user_behaviors 的 report 类型，但举报后的处理流程完全空白：
- 举报达到多少次触发人工审核？
- 举报是否通知被举报者？
- 举报内容是否对举报人可见（防止恶意举报）？

> 建议：MVP 阶段简化为"举报后进入待审核队列，由 editor/admin 人工处理"，不做自动化。请 PM 确认。

**问题 2：`/search` 接口的 `keyword` 是否必填？**

当前设计标注为必填。但如果用户进入搜索页不输入关键词，是否展示热门内容或最新内容？这涉及前端搜索页的空状态设计。

> 建议：keyword 必填，空状态由前端引导用户输入，不走搜索接口。请 PM 确认。

---

### 🟡 P1 - 影响 P1 功能开发

**问题 3：积分变更是否需要明细记录？**

当前设计积分只存在 users.points 字段，没有积分流水表。用户无法查看"为什么我的积分是这个数"。

> 建议：如果 P1 阶段要做"我的积分"页面，需要补充 `point_logs` 表。如果 MVP 不展示积分明细，可以后置。请 PM 决策是否纳入 P1。

**问题 4：评论的点赞是否纳入 user_behaviors？**

当前 user_behaviors 只关联 article_id，评论点赞（comments 表有 like_count）没有对应的行为记录。

> 建议：MVP 阶段评论点赞不做（需求文档 P1 互动功能中也未明确提及评论点赞），后续扩展时 user_behaviors 增加 `target_type` 字段区分文章/评论。请 PM 确认评论点赞是否在 P1 范围内。

## 四、开发侧风险提示

1. **zhparser 镜像稳定性**：`abcfy2/zhparser:14` 是社区维护镜像，非官方。建议在 CI 中固定镜像 digest，避免上游变更导致构建不一致。

2. **user_behaviors 表写入量**：浏览行为（view）高频写入，单机 PostgreSQL 在内容量大后可能成为瓶颈。建议 view 行为先写 Redis，异步批量落库，或仅统计 UV 不记录每次浏览。**此决策影响表结构，需在 Week 5 后端开发前确认。**

3. **搜索接口 `GET /search` 与列表接口 `GET /articles` 的 keyword 参数**：两个接口都支持关键词，职责需要在前端层面明确区分，避免用户困惑。建议：列表接口的 keyword 做简单标题过滤（LIKE），搜索接口才走全文检索。
